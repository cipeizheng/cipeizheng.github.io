跳跃游戏 I 和 II#

在 LeetCode 上有两个跳跃游戏的题目，分别是 Jump Game I 和 Jump Game II。这两个题目都涉及到如何在一个数组中跳跃到最后一个元素，而且都用到了贪心算法的思路。这两个问题的描述几乎如出一辙：给定一个非负整数数组，每个元素代表你能跳跃的最大长度，判断你是否能从起点到达终点。然而，尽管它们的表面描述相似，但它们的解法却截然不同。

  • 跳跃游戏 I 问的是：你是否能到达最后一个下标？

  • 跳跃游戏 II 问的是：到达最后一个下标的最小跳跃次数是多少？

跳跃游戏 I (canJump)：探索可能性的边界#

我们先来看问题相对简单的一方：跳跃游戏 I。它的目标是回答一个“是”或“否”的问题。

问题描述 (#55. Can Jump) 给你一个非负整数数组 nums，你最初位于数组的第一个下标。数组中的每个元素代表你在该位置可以跳跃的最大长度。判断你是否能够到达最后一个下标，如果可以，返回 true；否则，返回 false。

要解决这个问题，我们只需要找到任何一条可以到达终点的路径即可。这引导我们采用一种目光长远的贪心策略：在任何时刻，我只关心我能到达的最远“势力范围”有多大。

下面是一个非常高效且优雅的贪心解法：

class Solution {
    public boolean canJump(int[] nums) {
        // cover 代表从起点开始，我们能够到达的最远位置的索引
        int cover = 0;
        if (nums.length == 1) {
            return true;
        }

        // 循环条件是关键: i <= cover
        // 我们的“脚步” i，必须在我们已知的、能够到达的范围之内
        for (int i = 0; i <= cover; i++) {
            // 在当前位置 i，尝试更新最远覆盖范围
            cover = Math.max(cover, i + nums[i]);

            // 如果覆盖范围已经到达或超过了终点，任务完成
            if (cover >= nums.length - 1) {
                return true;
            }
        }

        // 如果循环正常结束，说明我们的脚步 i 跑出了 cover 的范围
        // 意味着前方出现了无法跨越的“鸿沟”，无法到达终点
        return false;
    }
}

这个算法的精髓在于 cover 变量和 for (int i = 0; i <= cover; i++) 这个循环条件。

  1. cover：最远的边界

    cover 变量维护的是一个动态的、不断扩张的“边界”。它记录了从索引 0 出发，综合所有已经访问过的、可达的位置的跳跃能力后，我们理论上能触及的最远索引。

  2. 贪心选择：不断扩张

    在循环的每一步 i，我们都做出一个贪心的选择：cover = Math.max(cover, i + nums[i])。我们利用当前位置 i 的跳跃能力 nums[i]，去尝试更新我们的最远边界 cover。我们不关心是哪一步具体让我们跳得最远，我们只关心这个最远边界是否在扩张。

  3. i <= cover：脚踏实地

    这个条件是算法正确性的基石。它确保了我们正在检查的索引 i 始终是可达的。想象一下，如果 cover 停止增长，而 i 持续递增，一旦 i 变得比 cover 大，就意味着索引 i 位于我们已知可达范围之外，我们永远无法踏上那一步。此时循环自然终止，正确地返回 false。

打个比方：这就像在地图上扩张领土。cover 是你领土的边界。我的侦察兵 i 只能在已有的领土上移动 (i <= cover)。每到一个新的地方，侦察兵就会向远处眺望，如果能看到更远的地方，就更新领土的边界 (cover = Math.max(...))。我的目标只有一个：判断我的领土边界能否最终覆盖目的地。我根本不关心侦察兵走了几步，或者换了多少个瞭望塔。

跳跃游戏 II (jump)：寻找最优化的路径#

现在，我们把难度稍微提升，进入跳跃游戏 II。问题变了，它不再满足于“能到就行”，而是追求“怎样到最省事”。

问题描述 (#45. Jump Game II) 给你一个非负整数数组 nums … 返回到达 nums[n - 1] 的最小跳跃次数。

这个问题要求的是一个最优解（最小值），因此，前面那种只顾扩张边界、不计成本的策略就不再适用了。我们需要一种新的、更加“步步为营”的贪心策略。

下面是跳跃游戏 II 的标准贪心解法：

class Solution {
    public int jump(int[] nums) {
        int jumps = 0;            // 记录跳跃的次数
        int current_jump_end = 0; // 当前这一跳所能到达的最远边界
        int farthest = 0;         // 在当前跳跃范围内，能展望到的下一步的最远边界

        // 我们遍历到倒数第二个元素即可，因为到达最后一个元素时无需再跳
        for (int i = 0; i < nums.length - 1; i++) {
            // 在[0...i]的范围内，更新下一步能跳到的最远距离
            farthest = Math.max(farthest, i + nums[i]);

            // 这是算法的核心决策点
            // 当 i 到达了上一次跳跃的边界时，我们必须进行下一次跳跃
            if (i == current_jump_end) {
                jumps++; // 跳跃次数加一
                current_jump_end = farthest; // 更新下一次跳跃的边界

                // 优化：如果新边界已经可以覆盖终点，可以提前结束
                if (current_jump_end >= nums.length - 1) {
                    break;
                }
            }
        }
        return jumps;
    }
}

与问题 I 不同，这里我们引入了三个关键变量来协同工作：

  1. jumps：成本计数器

    这是我们的最终目标，直接记录跳跃的次数。

  2. current_jump_end：当前步的终点

    这是区分“这一步”和“下一步”的关键。它标记了我们当前所处的这次跳跃，最多能覆盖到哪个索引。

  3. farthest：下一步的起点

    它和问题 I 中的 cover 有点像，都是在寻找最远距离。但它的角色变了：它是在为下一次跳跃做规划。在当前跳跃的范围内（从上一个 end 到 current_jump_end），我们利用所有这些位置去寻找一个能让下一次跳跃走得最远的位置，并将这个最远距离记录在 farthest 中。

  4. if (i == current_jump_end)：决策时刻

    这是整个算法的灵魂。当 i 触及 current_jump_end 时，意味着“当前这一跳的能量已经用完”，我们已经走到了这次跳跃能到达的最远点。此时，我们别无选择，必须进行一次新的跳跃。于是 jumps 加一，同时，我们把之前规划好的 farthest 赋值给 current_jump_end，作为新一轮跳跃的边界。

打个比方：这次我是在玩一个平台跳跃游戏。我每次起跳（jumps++）后，可以在空中滑行一段距离（直到 current_jump_end）。在滑行的过程中（for 循环），我并没有消耗新的跳跃次数，而是在紧张地观察，寻找下一个最佳的落脚点，这个落脚点要能让我下一次跳得最远（更新 farthest）。当我滑行到本次跳跃的极限时（i == current_jump_end），我必须再次起跳，开始下一段滑行。

四、总结与对比#

现在，我们可以清晰地看到两种思路的根本差异了。我们可以用一个表格来总结：

特征跳跃游戏 I (canJump)跳跃游戏 II (jump)
问题本质可能性问题 (Possibility)最优化问题 (Optimization)
核心提问“是否能到达终点？” (Yes/No)“最少需要几步才能到达终点？” (Minimum number)
贪心策略尽力扩张最远覆盖范围。在当前跳跃范围内寻找下一次跳跃的最佳起点。
关键变量cover (一个统一的最远边界)jumps, current_jump_end, farthest (步数、当前步边界、下一步边界)
决策时刻随时更新最远距离，只要达标就成功。只有到达当前步的边界时，才决策并更新步数。
思想类比连续覆盖：我的势力范围能否连续不断地延伸到终点？分步/分层覆盖：我最少需要跨越几个“层次”才能到达终点？ (类似BFS)

当然，这两个问题也是有共同点的，它们都可以用贪心算法来解决，都需要关注“覆盖范围”，但它们的侧重点不同。
