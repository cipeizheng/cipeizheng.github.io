DIMM 基本原理与架构

DIMM 简介 DIMM（Dual Inline Memory Module，双列直插式内存模块），简单来说，就是PC里插的内存条，观察内存条，他的接口两侧都有金手指，这就是“双列”的含义。在内存条上，有很多方形的小芯片，这些芯片就是实际存储数据的DRAM芯片。 DIMM的设计允许每个引脚在模块的两侧都有连接，这样可以提供更高的引脚数量和更大的数据传输带宽（相较于SIMM，单列直插式内存模块）。接下来我们介绍一下 DIMM 的层级结构。 DIMM 层级结构 DIMM 的层级结构可以分为以下几个层次： DIMM：这是最顶层的结构，包含多个 DRAM 芯片。DIMM 模块通过金手指与主板上的内存插槽连接。 Rank：DIMM 可以包含一个或多个 Rank。每个 Rank 是一组 DRAM 芯片，通常是 8 个或 16 个芯片，它们共同工作以提供一个数据总线宽度（一般来说是 64 位，如果是 ECC 内存，则是 72 位）。每个 Rank 可以独立地进行读写操作。但是，内存控制器在同一时间只能访问一个 Rank。 Chip：每个 Rank 由多个 DRAM 芯片组成。每个芯片通常有一个数据宽度、一般分为x4、x8、x16等类型。x8 芯片表示每个芯片有 8 位数据宽度，那么 8 个 x8 芯片就可以组成一个 64 位的数据总线，也即一个 Rank。 Bank Group：每个 DRAM 芯片内部通常包含多个 Bank Group。Bank Group 是 DRAM 内部的一个逻辑划分，用于提高并行访问能力。每个 Bank Group 包含多个 Bank。DDR4 通常为 2 或 4 个 Bank Group（每个含 4 个 Bank，共 8/16 个 Bank）；DDR5 扩展到 8 个 Bank Group（每个含 4 个 Bank，共 32 个 Bank） ...

Java 泛型的通配符与PECS原则

Java泛型中的extends与super Java在泛型中引入extends和super通配符，其核心目的是为了在保证类型安全的前提下，增加泛型代码的灵活性，解决泛型类型固有的“不可变性”（Invariance）所带来的限制。而**PECS原则（Producer Extends, Consumer Super）**则是对如何正确使用这两种通配符的高度概括和实用总结。 在Java中，泛型类型默认是不可变的。这意味着，即使String是Object的子类，List<String>也不是List<Object>的子类。 让我们来看一个例子： // 假设我们有一个打印Object列表的方法 public void printList(List<Object> list) { for (Object obj : list) { System.out.println(obj); } } // 现在我们有一个String列表 List<String> strList = new ArrayList<>(); strList.add("Hello"); strList.add("World"); // 下面的调用将会导致编译错误！ // printList(strList); // 编译不通过 这个编译错误可能会让初学者感到困惑。编译器之所以阻止这种行为，是为了防止潜在的类型安全问题。如果printList(strList)是合法的，那么在printList方法内部，我们就可以向这个列表中添加任何Object类型的对象，例如一个Integer： public void printList(List<Object> list) { list.add(123); // 合法，因为123是Object的子类 for (Object obj : list) { System.out.println(obj); } } 这样一来，当我们回到调用方，尝试从原来的strList中取出一个元素时，就可能会遇到ClassCastException，因为我们期望得到一个String，但实际上取出的可能是一个Integer。例如下面的代码： List<String> strList = new ArrayList<>(); strList.add("Hello"); strList.add("World"); printList(strList); // 假设这行代码编译通过了，那么strList的内容是["Hello", "World", 123] String s = strList.get(3); // 这里会抛出ClassCastException 于是，Java的泛型设计在编译期就杜绝了这种风险，禁止将List<String>传递给List<Object>类型的方法。 ...
